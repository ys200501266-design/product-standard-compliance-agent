# 产品经理项目复盘

## 1. 为什么选择这个项目

我选择“产品标准合规推荐助手”，是因为它比普通商品推荐更能体现 AI 产品经理的判断力。普通推荐关注商品相关性，而这个项目关注标准、证据、风险和可解释性。

## 2. 为什么它不是普通商品推荐

普通商品推荐通常围绕价格、销量、评价和品牌。这个项目先查询标准，再判断商品是否公开标注相关执行标准或认证信息。推荐不是从“看起来不错”出发，而是从“证据是否足够”出发。

## 3. 用户需求如何结构化

用户自然语言会被拆成：

| 字段 | 作用 |
|---|---|
| product_name | 确定检索对象 |
| use_case | 判断适用场景 |
| budget | 判断价格合理性 |
| region | 判断标准和购买地区 |
| extra_requirements | 判断安全、材质、防漏等附加条件 |

## 4. 为什么要先查标准再推荐商品

如果直接搜索商品，系统只能找到“相关商品”。先查标准可以建立判断框架：哪些标准可能适用、哪些信息必须确认、哪些商品信息只是营销描述。这样能降低误推荐风险。

## 5. 如何降低 LLM 幻觉

我在 Prompt 中加入了硬约束：

- 标准编号必须来自搜索结果。
- 商品链接必须来自搜索结果。
- 检测报告不能编造。
- 无证据时必须输出“暂无公开证据证明”。
- 区分事实、推断和不确定性。

## 6. 如何设计证据等级

证据等级分为 A-E。A 级用于确认标准来源，B/C 级用于判断商品公开标注，D 级只能作为线索，E 级不允许强推荐。这个模型让推荐结果具备可解释性。

## 7. 如何设计评分规则

评分使用 100 分制，标准匹配度占 40 分，证据可信度占 25 分。这样设计是为了突出项目定位：它不是低价推荐工具，而是合规初筛工具。

## 8. 如何处理搜索失败和证据不足

搜索失败时不强行输出推荐，而是进入兜底报告。标准不足时说明“本次结果仅能作为初步搜索线索”。商品证据不足时说明“不输出强推荐商品”。

## 9. 当前版本的局限

- 搜索 API 的覆盖范围会影响结果。
- 电商详情图需要 OCR 才能完整识别。
- 标准适用性仍需要人工二次确认。
- 当前版本不能替代专业检测或法律意见。

## 10. 如果进入真实产品阶段，下一步怎么做

1. 接入稳定搜索 API 和网页抓取。
2. 建立标准知识库。
3. 增加 OCR 和截图留存。
4. 引入人工审核节点。
5. 对高风险品类做白名单和强制复核。

## 11. 面试官可能追问的问题和回答

### Q1：为什么不直接用普通搜索推荐商品？

A1：普通搜索只能说明商品“相关”，不能证明它“符合标准”。这个项目先建立标准约束，再做商品证据判断，能降低误推荐风险。

### Q2：怎么防止 LLM 编造标准？

A2：标准编号必须来自搜索结果或标准平台来源。如果搜索结果中没有标准编号，Prompt 要求输出“暂无公开证据证明”，不能根据常识补全。

### Q3：为什么要设计证据等级？

A3：因为不同来源的可信度不同。政府标准平台和品牌检测报告不能和第三方导购文章放在同一层级。证据等级能让推荐更可解释。

### Q4：为什么电商详情页不是最高等级证据？

A4：电商详情页可能存在营销表达或信息不完整。它可以作为商品标注线索，但购买前仍需要看品牌官网、检测报告或包装标识。

### Q5：如果搜索不到标准怎么办？

A5：不强行推荐。系统会返回兜底报告，说明未找到足够公开标准信息，本次结果只能作为初步线索。

### Q6：评分权重为什么这样设计？

A6：项目目标是合规初筛，所以标准匹配度和证据可信度合计 65 分。价格和品牌重要，但不能超过合规证据。

### Q7：这个项目怎么产品化？

A7：可以接入标准知识库、网页抓取、OCR、人工审核和报告导出，扩展到企业采购或垂直导购工具。

### Q8：如何验证这个项目真的有效？

A8：可以用多个品类做测试，记录标准召回率、证据完整度、幻觉控制率和用户是否能理解推荐理由。

### Q9：它和普通 RAG 有什么区别？

A9：普通 RAG 强调检索增强回答，这个项目强调工作流决策：品类识别、标准检索、证据分级、评分排序和异常兜底。

### Q10：你在项目里体现的产品能力是什么？

A10：主要是需求拆解、流程设计、证据建模、Prompt 风控、异常处理和作品集表达。
